# CMS and Messages Setup

This document explains the contact/message modal, Supabase message storage plan, admin message dashboard, Decap CMS setup, and remaining production work.

## 1. Why GitHub Pages Alone Is Not Enough

GitHub Pages can host static files, but it cannot securely:

- store private contact messages
- read visitor IP addresses
- enforce real rate limits
- protect admin pages with server-side authentication
- keep Supabase service keys or other secrets private

The current site can show the message modal on GitHub Pages, but real message storage and spam protection require a backend or hosted service.

## 2. Prepared Backend Option

The prepared backend option is Supabase.

The contact modal in `js/main.js` submits to a Supabase Edge Function:

```txt
https://ufcvdlidsdrcdnjswocj.supabase.co/functions/v1/submit-message
```

The Edge Function stores messages in:

```txt
public.contact_messages
```

Full Supabase setup details live in:

```txt
docs/SUPABASE_SETUP.md
```

Do not put a Supabase service role key or `sb_secret_...` key in this static website. Only the public publishable key belongs in browser code.

## 3. How the Contact Modal Works

Links with this attribute open the modal:

```html
data-message-modal-open
```

The modal is generated by `js/main.js` and includes:

- Name
- Company
- Contact Mail
- optional phone number with country code selector
- message textarea with a 2000 character limit
- character counter
- Send button
- Close button
- honeypot field
- ESC-to-close behavior
- basic focus trap

The homepage hero `Email Newsletter` button opens the modal. Sidebar `Email signup` links stay as `mailto:balabanenes111@icloud.com`.

## 4. Spam Protection

Frontend protections already included:

- required Name, Contact Mail, and Message validation
- email format validation
- optional phone number format validation
- message maximum length of 2000 characters
- meaningful-message checks for very short text, repeated characters, low word count, low vowel ratio, random symbols, and low unique character count
- honeypot field
- timestamp check to reject instantly submitted bot forms
- localStorage cooldown after a successful send

Backend/server-side protections prepared in `supabase/functions/submit-message/index.ts`:

- Supabase Edge Function validation
- IP hashing before storage
- repeated-message rate checks by IP hash
- spam scoring and spam status

Backend/server-side protections still required before production:

- Supabase Row Level Security policies
- function secrets and deployment
- optional Cloudflare Turnstile or reCAPTCHA verification

A static GitHub Pages-only deployment cannot do true IP-based rate limiting by itself.

## 6. Admin Authentication

Do not expose private messages through a public static page.

Production admin access needs real authentication. Recommended options:

- Supabase Auth with protected Row Level Security policies
- Netlify Identity plus a serverless function that reads messages securely
- Cloudflare Access in front of the admin route

For Supabase, create policies so public users can insert messages but cannot select/read them. Only authenticated admin users should be able to read messages.

## 7. CMS Setup

The project includes Decap CMS:

```txt
admin/cms.html
admin/config.yml
```

Content is stored as JSON in the repository:

```txt
content/
  notes/notes.json
  projects/projects.json
  illustrations/illustrations.json
  minigames/minigames.json
```

The CMS is configured for:

- Notes
- Projects
- Certificates
- Illustrations
- Minigames

The public pages load these JSON files with vanilla JavaScript in `js/main.js`.

For production, configure Decap CMS GitHub authentication, then visit:

```txt
/admin/
```

For local CMS testing, Decap CMS supports `local_backend: true`, but it requires the Decap local backend service from the Decap CMS tooling.

Detailed CMS setup is documented in:

```txt
docs/DECAP_CMS_SETUP.md
```
